Dynamic vs. Static typing

People keep talking about this like we should know what it is.

11/22/2015

Typecasting

Ok, I'm going to confess to this now: I really thought that when people referred to Ruby and JavaScript as "dynamically typed languages", it had to do with how characters were put into the computer somehow. Like, how they were typed. With your keyboard.


I didn't know why, exactly, this would matter, or what it would mean, but it was the only thing I could think of. So I was pretty thrilled to actually look it up. Of course, like everything else on the internet, I found mostly arguments.


The basics: the "types" referred to in dynamic and static types are data types. There's a second bit to this: besides "dynamic" and "static," there's also "weak" and "strong" types, which are actually something we've had the chance to compare.


What's a type system?

In order to understand dynamic vs. static typing, we have to understand a type system. When languages assign a "type" to data, they are giving the data--which at the base is just a set of bits--meaning. For example, "Types" in Ruby are strings, integers, floats, and booleans. Types in JavaScript are strings, numbers, and booleans.
A type system allows a computer to catch errors like trying to multiply 5, say, by "purple". 5 is an integer type, and "purple" is a string, so being able to issue an error immediately saves valuable time and energy and stops the computer from making a nonsense output.


Dynamic vs. Static types

Both Ruby and JavaScript are dynamically typed languages. In a dynamically typed language, once a variable is assigned to data of one time, it can be changed later to data of another type. For example, let us consider unfortunate convict Jean Valjean. When Jean Valjean is in the world, who_am_i = "Jean Valjean". But when Jean Valjean is thrown in prison, who_am_i = 24601. Such an operation (changing a string to an integer) is legal in a dynamically typed language, but would throw an error in a statically typed language, which does not allow changing a variable's data from one type to another.


This is the internet, so of course there's an argument. Static typed languages like C, C++, Java, Scala, Haskell, and, yes, Fortran (along with many more) are harder to program initially but because they catch errors right away, they have less of a chance of human error causing a bug; a dynamic typed language will try much harder to run a typo than a static typed language.
Dynamic typed languages like Ruby, JavaScript, PHP, Perl, Smalltalk, Lisp, and more, can be easier to program in because they create more elastic, forgiving programs. The internet seems to mostly think that prototypes and small things that need to change a lot should be done in dynamic-typed languages, and big, heavy systems with a need for stability over development speed should be done in static-typed systems.
The internet was much less civil in expressing that opinion.


Strong vs. weak types (a bonus): Although we have only worked with dynamic-typed languages, we have worked with one strong-typed language--Ruby--and one weak-typed language: JavaScript. In strong typing, a variable's type cannot be coerced into another type: 3 is not divisible by "3" and will throw an error. But in weak typing, the language will attempt to perform the operation, and 3/"3" will generate 1.
